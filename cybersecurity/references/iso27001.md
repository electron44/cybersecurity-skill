---

## Vue d'ensemble ISO 27001:2022

ISO 27001:2022 définit les exigences pour un **Système de Management de la Sécurité de l'Information (SMSI)**.
La version 2022 contient **93 contrôles** répartis en **4 thèmes** (contre 114 contrôles en 14 domaines pour la version 2013).

```
Thème A.5  — Contrôles organisationnels (37 contrôles)
Thème A.6  — Contrôles humains (8 contrôles)
Thème A.7  — Contrôles physiques (14 contrôles)
Thème A.8  — Contrôles technologiques (34 contrôles) ← Le plus pertinent pour le dev
```

## Mapping skill ↔ ISO 27001:2022

### A.8 — Contrôles technologiques (priorité développement)

#### A.8.1 — User endpoint devices
- Politiques de sécurité des postes de travail
- Chiffrement des disques (BitLocker, FileVault)
- MDM pour les appareils mobiles

#### A.8.2 — Privileged access rights
→ **Couche 5 du skill** (Auth/Authz)
- Revue trimestrielle des droits privilégiés
- Comptes admin dédiés (séparés des comptes courants)
- MFA obligatoire pour les accès admin

#### A.8.4 — Access to source code
- Contrôle d'accès au dépôt de code (Git)
- Branch protection rules (review obligatoire)
- Audit trail des modifications de code

#### A.8.7 — Protection against malware
- Antivirus/EDR sur tous les endpoints
- Filtrage des emails (anti-phishing)
- Politique de gestion des médias amovibles

#### A.8.8 — Management of technical vulnerabilities
→ **Couche 2 du skill** (Dépendances)
- Processus de veille CVE documenté
- SLA de correction selon criticité :
  - CRITIQUE : 24h
  - HAUTE : 7 jours
  - MOYENNE : 30 jours
  - BASSE : 90 jours
- Scan de vulnérabilités régulier (DAST/SAST)

#### A.8.9 — Configuration management
→ **Couche 4 du skill** (Infrastructure)
- Baseline de configuration documentée
- Configuration as Code (Terraform, Ansible)
- Audit de conformité automatisé

#### A.8.10 — Information deletion
- Politique de rétention des données
- Procédure de suppression sécurisée (NIST SP 800-88)
- Logs de destruction des données sensibles

#### A.8.11 — Data masking
- Pseudonymisation des données de test
- Ne jamais utiliser de données de production en dev/staging
- Masquage des données sensibles dans les logs

#### A.8.12 — Data leakage prevention (DLP)
- Monitoring des transferts de données sortants
- Classification automatique des documents
- Contrôle des partages cloud (Google Drive, OneDrive)

#### A.8.15 — Logging
→ **Couche 7 du skill** (Observabilité)
- Logs d'accès aux systèmes critiques : 12 mois minimum
- Logs d'audit immuables (append-only)
- Synchronisation NTP pour horodatage fiable

#### A.8.16 — Monitoring activities
- SIEM configuré avec règles d'alerte
- Revue des alertes de sécurité : quotidienne
- Rapport mensuel de sécurité pour la direction

#### A.8.20 — Networks security
- Segmentation réseau (DMZ, VLAN)
- Firewall avec règles documentées et revues
- IDS/IPS sur le périmètre

#### A.8.22 — Segregation of networks
- Séparation réseau dev / staging / production
- Isolation des systèmes critiques
- Filtrage du trafic inter-zones

#### A.8.23 — Web filtering
- Filtrage des URLs malveillantes
- Contrôle des téléchargements
- Proxy HTTPS avec inspection SSL

#### A.8.24 — Use of cryptography
→ **Couche 2 & 5 du skill** (Crypto)
- Politique de cryptographie documentée
- Inventaire des algorithmes utilisés
- Gestion du cycle de vie des clés (génération, rotation, révocation)

#### A.8.25 — Secure development lifecycle
**C'est le contrôle le plus important pour un développeur :**
- Sécurité intégrée dès la conception (Security by Design)
- Threat modeling pour chaque nouvelle feature
- Code review de sécurité obligatoire
- Tests de sécurité (SAST, DAST, pentest) avant release
- Formation sécurité des développeurs

#### A.8.26 — Application security requirements
- Exigences de sécurité documentées avant développement
- Critères d'acceptance de sécurité définis
- Tests de pénétration annuels minimum

#### A.8.27 — Secure system architecture and engineering principles
- Architecture Zero Trust
- Defense in depth
- Principe du moindre privilège appliqué à l'architecture
- Séparation des responsabilités

#### A.8.28 — Secure coding
→ **Couches 1–6 du skill** (toute l'analyse statique)
- Standards de codage sécurisé documentés (ex : OWASP Secure Coding Practices)
- SAST intégré en CI/CD
- Formation OWASP pour les développeurs
- Pas de secrets dans le code source

#### A.8.29 — Security testing in development and acceptance
- Tests unitaires de sécurité (cas négatifs)
- DAST automatisé avant chaque release
- Pentest externe annuel
- Bug bounty program si applicable

#### A.8.30 — Outsourced development
- Contrats avec clauses de sécurité (sous-traitants développement)
- Audit de code des livrables externes
- Accès limité au code source pour les prestataires

#### A.8.31 — Separation of development, test and production environments
- Environnements strictement séparés
- Données de prod jamais copiées en dev
- Processus de déploiement documenté et audité

#### A.8.32 — Change management
- Toute modification → ticket + approval + test + rollback plan
- Emergency change process documenté
- Post-deployment verification

#### A.8.33 — Test information
- Données de test anonymisées/pseudonymisées
- Politique de gestion des données de test
- Suppression des données de test après usage

#### A.8.34 — Protection of information systems during audit testing
- Tests de sécurité planifiés et approuvés
- Scope de test formellement défini
- Isolation de l'environnement de test pendant les audits

## A.5 — Contrôles organisationnels (sélection pertinente)

#### A.5.7 — Threat intelligence
- Abonnement à des feeds de threat intelligence
- Processus de traitement et diffusion de l'information
- Intégration dans les processus de réponse à incident

#### A.5.23 — Information security for use of cloud services
- Politique d'utilisation des services cloud
- Évaluation de sécurité des providers cloud (AWS, GCP, Azure)
- Contrôle des données stockées hors périmètre

#### A.5.29 — Information security during disruption
→ Lien avec NIST CSF RECOVER
- Plan de continuité intégrant la sécurité
- Tests réguliers des procédures de reprise

#### A.5.30 — ICT readiness for business continuity
- RTO et RPO définis pour chaque système critique
- Tests de reprise documentés et effectués
